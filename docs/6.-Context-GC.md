# コンテキストGC仕様書

本文書は Li+ プログラムにおけるコンテキスト劣化対策としてのサブエージェント委任の仕様を定義する。

---

## 目的

長時間セッションにおいて operations 層のルール遵守が劣化する問題を解決する。

---

## 前提

- re-read hook は正常に発火し、Li+operations.md は読み込まれている
- 読んだ上で無視される（注意の薄まり）
- Character_Instance は system-reminder 格の毎ターン注入で維持できるが、operations 層は tool result として読み込まれるため注意重みが低い
- セッションが長くなるほど、作業コンテキストが蓄積し、ルールの相対的重みが低下する

### 原因構造

1. **注意の薄まり** — 周囲ノイズの増加で tool result 内のルールへの注意重みが下がる
2. **注入ポイントの格差** — system prompt > system-reminder > tool result。operations は tool result に入る
3. **ゴミの蓄積** — LLM は不要コンテキストを能動的に削除できない（構造的制約）

---

## 解決アプローチ: コンテキスト駆動エージェント分離

動機 = コンテキスト最適化、手段 = エージェント分離、基盤 = GitHub。
サブエージェント機能は最適化であり必須ではない。ない環境では従来通り動く。
API名やランタイム固有の呼び出し方は仕様に含めない（AIは自分のツールを知っている）。

### メインエージェント（Model Layer + Task Layer）

- Li+core.md + Li+github.md = 常駐
- 人間との対話 = 要件蒸留
- issue管理 = 対話から生まれる設計判断
- Character_Instance = 対話品質
- サブの成果物レビュー = pushの判断はメインの責務

### サブエージェント（Operations Layer 実行ワーカー）

- Li+operations.md を毎回新鮮なコンテキストで読む
- 毎回生まれたてのコンテキストだから注意の薄まりがない
- モデルは同一（能力差なし）
- 成果物 = 仕様 + 実装 + テスト（三つで一つの変更単位、分離しない）

### メインからサブへ渡す情報

- operations.md のパス（読ませる）
- issue番号、リポジトリ、ブランチ名
- 編集スコープ（何を触っていい/だめ）
- コミットするがpushしない（メインがレビュー後にpush）

### トリガー定義

委任対象はpre-pushフェーズ（ファイル変更を伴う作業）に限定する。
post-pushフェーズ（remote state操作）はメインが担当する。

pre-push（on_branch, on_commit — 委任可能）:
- サブエージェント機能が使える場合 → サブに委任。メイン自身は operations.md を読まない
- サブ完了後 → メインが diff 確認・レビュー → push
- 使えない場合 → 従来通りメインが読んで実行

post-push（on_pr, on_ci, on_review, on_merge, on_release — メイン担当）:
- remote state の観測・判断・実行が本体のため委任しない
- メインが operations.md の該当セクションを読んで実行

---

## 制約

- 環境非依存であること（Claude Code / Codex / 将来のランタイム）
- Li+ の層構造と矛盾しないこと
- 仕様・実装・テストの三つ一組は分離しない（循環依存のため）
- ユーザーのワークスペース構成に制約を課さない
