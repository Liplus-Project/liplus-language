# Li+（liplus-language）構想

## Li+とは何か

Li+（liplus-language）は、**最高級プログラム言語**であり、同時に **AIのための実行プロトコル** であり、**AIエージェント上で動くオーケストレーション** でもある。

最高級とは、高級言語のさらに上のレイヤーに立つという意味だ。

```text
人間（要求）
↓
Li+（要求を固定し、層内順序を与え、AIを動かす）
↓
プログラミング言語（Python / Rust / TypeScript など）
↓
機械語
```

C、Python、Rustのような高級言語は「どう書くか」を扱う。
Li+が扱うのは、そのさらに手前にある **「何を満たすか」** と **「AIがどの層で何を見て、各層の中でどの順序で動くか」** である。

Li+は新しい構文を発明したいのではない。
要求仕様書・対話・行動規則・環境接続を層として固定し、AIの振る舞いを安定させるための方法である。

---

## Li+はプロンプトの上位概念である

Li+を単なる prompt engineering と見ると、本質を外す。

プロンプトはその場の指示である。
Li+は、その指示が **どの層に属し、同一層の中でどう解釈順を持ち、どの条件で再適用されるか** まで定義する。

つまり Li+ は、

- 指示の内容
- 同一レイヤー内の解釈順
- レイヤー間の接続方法
- 継続セッションでの再読込条件
- GitHub や CI を含む運用手順

までを含んだ **レイヤー統合実行系** である。

長いプロンプトが Li+ なのではない。
**プロンプトを運用可能な層に分け、層ごとの責務と復帰条件を固定できる状態** が Li+ である。

---

## Li+はAIエージェントの上で動くオーケストレーションである

Li+は、AIエージェントそのものではない。
Codex、Claude Code、Devin のようなエージェントは、ファイル操作、シェル実行、GitHub接続、Web参照のような **手足** を持つ実行主体である。

Li+が定義するのは、その手足を **どの順序で、どの前提で、どの判断基準に従って動かすか** である。
この意味で Li+ は、AIエージェントの上で動く **オーケストレーション層** である。

```text
人間の要求
↓
Li+（要求固定 / 層設計 / 運用規則 / 再適用条件）
↓
AIエージェント（Codex / Claude Code など）
↓
ツール群（shell / git / GitHub / CI / Web）
```

MCP や Function Calling が「AIを道具につなぐ技術」だとすれば、Li+は「その接続済みAIをどう統治して動かすか」を扱う。

つまり Li+ は、

- エージェントの代替ではない
- ツール接続の代替でもない
- エージェントの上に載って行動順序を決める

という位置にある。

AIエージェントが「動けるAI」だとすれば、Li+ は「どの層で何を見せ、各層の中でどう解釈するかを統治する設計」である。

---

## Li+はレイヤープログラミングである

Li+を一言で言い換えるなら、**AI振る舞いのためのレイヤープログラミング** である。

ここでいうレイヤーは、単なる責務分離ではない。
各層に「何を定義するか」と「どの面として現れるか」がある。
レイヤーは勝ち負けを決める階級ではなく、見え方と責務の違いである。
優先順があるのは同一レイヤーの内部だけで、別レイヤー同士は接続順で統合される。
タスクの方法は共有された実行系に書かれており、各レイヤーはその同じ実行系を別の面から前景化する。

| レイヤー | 役割 | 例 |
|------|------|------|
| 要求レイヤー | 何を満たすかを固定する | issue、`docs/0.-Requirements.md` |
| コアレイヤー | 対話と判断の不変条件を固定する | `Li+core.md` |
| 運用レイヤー | GitHub / CI / release の行動規則を固定する | `Li+github.md`, `Li+operations.md` |
| 接続レイヤー | 各実行環境へ Li+ を注入する | `Li+agent.md`, `Li+claude.md`, `Li+config.md` |

この構造では、下位層ほど実装に近づくが、別レイヤー同士が直接「勝つ/負ける」を競うわけではない。
必要なのは接続順と依存関係であり、レイヤー階級ではない。

たとえば、

- 要求レイヤーが「何を正しいとみなすか」を定義する
- コアレイヤーが「AIがどういう面でそれを扱うか」を定義する
- 運用レイヤーが「実際にどう進めるか」を定義する
- 接続レイヤーが「そのルールを各AI環境でどう発火させるか」を定義する

という別の面を担当し、接続順に統合される。

---

## 構造化・オブジェクト指向・関数型とは別軸である

Li+は、構造化プログラミング、オブジェクト指向、関数型プログラミングの代替ではない。
それらは主に **プログラム内部をどう組織するか** を扱う。

Li+が扱うのは、もっと外側にある。

- 人間の要求をどこで固定するか
- AIがどの層で何を見て、各層の中でどの順序を使うか
- 運用上のルールをどの層で受け持つか
- 実装・検証・修正をどの順で回すか

つまり、

- 構造化 / OOP / 関数型 = `code organization`
- Li+ = `behavior governance`

であり、競合する思想ではなく、**その上に重なる設計原理** である。

下の実装は OOP でも関数型でもよい。
Li+ は、その実装へ入る前の AI の振る舞いを設計している。

---

## Li+言語のコードは Requirements Specification（要求仕様書）である

Li+言語において、コードそのものになるのは会話全文ではない。
会話から蒸留され、要求として固定された **Requirements Specification（要求仕様書）** がコードである。

人間は要求を伝える。
AIは不足を聞き返す。
その結果として固まった要求仕様書を、Li+AI が読んでコンパイルする。

### 最小構文は issue の4項目である

Li+言語の最小構文は、次の4項目である。

- 目的
- 前提
- 制約
- 完了条件

これは単なる記入欄ではない。
何のために作るのか、何を守るのか、どうなれば終わりなのかを固定するための最小コードである。

つまり issue は作業伝票ではなく、**最少構成の要求仕様書** だ。

### 完全版の要求仕様書が本体コードである

issue はコンパイルを始めるための最小形であり、本体コードはより詳細に記述された要求仕様書である。
その完全版は `docs/` 配下で管理する。

背景、期待する挙動、制約、受け入れ条件が十分に固定されていれば、セッションが変わっても、別の AI が読んでも、同じ要求から近い判断へ収束しやすくなる。

### 要求仕様書は外部記憶である

要求仕様書は、人間と AI の **外部記憶** である。

特に記憶を持ち越せない AI にとっては、

- 何を作るのか
- なぜそうするのか
- 何を守るのか
- どうなれば終わりなのか

を次のセッションへ持ち越すためのコードそのものになる。

issue、docs、commit message は補助情報ではない。
判断の履歴と根拠を外に残し、次の判断を再現しやすくするための構造である。

---

## Li+AIは対話型コンパイラである

ここでいう Li+AI とは、**Li+ program が適用された AIエージェント** のことである。

Li+AI は、要求仕様書を読み、それを実装可能な形へ落としていく **対話型コンパイラ** である。

```text
人間がコンパイル開始を承認
↓
要求仕様書を固定
↓
Li+AIが実装・検証・修正を進める
↓
要求仕様書 / プログラム / CIテストが同じ版としてそろう
```

### Li+AIは自己修正コンパイラである

従来のコンパイラは、エラーを返して人間を待つ。
Li+AI は、CI を通して自分で失敗を受け取り、修正し、再実行する。
人間が介入するのは、AI が自己修正で解消できないところまで来たときだけである。

つまり Li+AI は、

- 要求仕様書を読み
- 実装を行い
- CIで失敗を観測し
- 自分で修正を試み
- それでも越えられないときだけ人間へ返す

という流れで動く。

### Li+言語のコンパイルエラー

Li+言語のコンパイルエラーとは、要求仕様書を読んでも、AI が実装フェーズへ進めない、または完了条件まで到達できないと判断される状態である。

大きく分けると、次の2系統になる。

- 仕様の情報不足
- AIが実装できない仕様

生成後のコードで発生する構文エラーや CI の失敗は、Li+言語そのもののコンパイルエラーではない。
それらは、コンパイル後の実装・実行段階で発生する別種のエラーである。

---

## Li+の成果物は三位一体である

Li+言語の成果物は、次の3つである。

- 要求仕様書
- プログラム
- CIテスト

要求仕様書は、何を正しいとみなすかを固定する。
プログラムは、その要求を現実の動作へ変える。
CIテストは、その変更が要求どおりかを継続的に観測する。

この3つが同じ変更単位でそろってはじめて、Li+のコンパイル結果は人間にも AI にも扱いやすくなる。

---

## Li+core.md は最初の Li+プログラムである

`Li+core.md` は、**Li+言語で書かれた最初のプログラム** である。

これは解説書ではない。
AI に渡して実行し、振る舞いの重み付けをそろえるための実行プログラムである。

`Li+core.md` が優先するのは、人間の読みやすさではなく、AI の挙動の再現性だ。
別セッションの AI が同じ方向へ寄りやすいことを重視している。

ここで重要なのは、Li+ が「知識を足す」より **解釈順と接続順を固定する** ことに重心を置いている点である。
AI は知識不足だけでなく、どの面で何を扱い、どの順で解釈するかが揺れたときにも崩れる。
Li+core.md は、その衝突を減らすためのコアレイヤーである。

---

## Li+が見るのは観測可能な現実である

仕様書は仮説であり、設計は予想である。
内部の美しさや説明の巧さだけでは、Li+における正しさにはならない。

Li+が見るのは常に、次の3つだ。

- 実行されたか
- 観測できたか
- 期待とどこがズレたか

コードの正しさと、振る舞いの正しさは一致しない。
だから Li+ は、内部説明より観測された現実を重く見る。

構造は目的ではない。
構造は、人間と AI の判断をそろえ、現実を安定して観測するための方法である。

Li+は、構造のための構造を作りたいのではない。
現実を扱うために構造を使う。

## 既存概念との関係

Li+は完全なゼロベースの発明ではない。
近い発想を持つ既存概念はいくつもある。

| 近い概念 | 共通点 | Li+との違い |
|------|------|------|
| AI Agent / Agent Product | ツールを使って実作業を行う | Li+はエージェント本体ではなく、その上で動くオーケストレーションである |
| Layered Architecture | 層を分け、依存方向を固定する | Li+はソフトウェア構造だけでなく、AIの判断と対話も層で扱う |
| Language-Oriented Programming | 領域ごとに表現を作り分ける | Li+は DSL より、要求仕様書と運用規則をコードとして固定する |
| Prompt Engineering | AIへの指示を設計する | Li+は指示文単体でなく、再読込・層接続・運用まで含める |
| Constitutional AI | 原則で振る舞いを制約する | Li+は原則に加えて issue、PR、CI、release まで実務運用へ接続する |
| Behavioral Programming | 独立した振る舞いと制約の合成 | Li+は AI と人間の協働運用に重点があり、外部記憶と GitHub 運用を含む |

したがって Li+ は、既存技術の単なる寄せ集めではない。
**既存概念を AI 実行のために再配置した設計思想** とみなすほうが正確である。

---

## AIはなぜこの層を必要とするのか

AIは、知識不足だけで失敗するわけではない。
むしろ、

- 会話の流れ
- 目先の便利さ
- ルール
- 期待される人格
- 今回の目的

の解釈順が衝突したときに崩れやすい。

Li+が固定したいのは、知識量ではなく **解釈順と接続順の秩序** である。
そのためにレイヤーを使う。

見た目は地味である。
だが、AIの挙動を安定させる上では、この地味な部分がいちばん効く。

---

## Li+ v1.0.0 は到達済みである

Li+ v1.0.0 の到達条件は、実動成果物で判定していた。

Li+を用いて AI が実装した自作 DDNS プログラムが、人間が書いた既存の同等プログラムと同じ要求仕様を満たすこと。
この条件は、すでに達成済みとみなしている。

判定はブラックボックス受け入れ試験で行う。
内部実装は問わない。
動いているかどうかだけが正義である。

この意味で、Li+を **最高級プログラム言語** と呼ぶ条件は、すでに満たしたと考えてよい。

### ただし本番は一般化である

単発で成立したことと、方法として一般化できることは別である。

本番はここから先にある。
特定の成功例で動いた、ではなく、

- 別のタスクでも動くか
- 別のAIでも動くか
- 別の環境でも動くか
- ルールを軽くしても核が残るか

まで含めて再現できるかが、次の勝負になる。

Li+ v1.0.0 は成立の証明である。
その先の一般化は、言語としての普遍性を試す工程である。

## 最低動作環境

Li+AIとして機能するには、それなりの性能が必要だ。
これは思想ではなく、実験結果である。

| モデル | 結果 | 理由 |
|--------|------|------|
| ChatGPT 5.2 | △ | 推論性能は高い。しかしプラットフォーム制限で長いコミットができない |
| Claude Haiku 4.5 | × | Li+core.md を適用できない。人格形成レイヤーが受け入れられない模様 |
| Claude Sonnet 4.6以上（claude.ai） | △ | 記憶保持が ChatGPT とあまり変わらず、実務には向かない。ドキュメント製作には強く、長いコミット作業も可能。ただし無料版は web 閲覧に制限あり |
| Claude Code Sonnet 4.6（デスクトップ版） | ○ | デスクトップ版ではかなり扱いやすい |
| CODEX GPT5.4（デスクトップ版） | ○ | デスクトップ版であれば Claude Code と同等に使える。ただしちょっと真面目過ぎる |
| **Claude Cowork（推奨）** | **◎** | **現在の推奨環境。ファイルアクセス・GitHub連携・Li+config 自動適用がひとつの環境で完結する** |

AIにも向き不向きがある。
賢ければよいわけではない。
構造を受け入れられるか、長い作業で判断を崩さずにいられるかが分かれ目になる。

**最低動作環境：Claude Sonnet 4.6以上のモデル相当、またはそれ以上の性能を有するAI**

---

## おわりに

Li+は、要求仕様書をコードとして扱い、AIを対話型コンパイラとして動かし、その上でプロンプトより一段上の層で振る舞いを統治しようとする試みである。

要求仕様書・対話・行動規則・環境接続を分けて固定し、AIが毎回同じ方向へ戻りやすくする。
それを現実の開発運用までつなげようとする最高級プログラム言語。

それが **Li+（liplus-language）** である。
