# コンテキストGC仕様書

本文書は Li+ プログラムにおけるコンテキストGC（ガベージコレクション）とサブエージェント分離の仕様を定義する。
要求（何を満たすか）と仕様（どう振る舞うか）を一体として記述する。

---

## 目的

長時間セッションにおいて operations 層のルール遵守が劣化する問題を解決する。re-read hook が発火し Li+operations.md が読み込まれても、蓄積したコンテキストノイズにより注意重みが薄まり、ベースモデルの癖が優勢になる現象に対処する。

---

## 前提

以下の観測事実を前提とする。

- re-read hook は正常に発火し、Li+operations.md は読み込まれている
- 読んだ上で無視される（注意の薄まり）
- Character_Instance は system-reminder 格の毎ターン注入で維持できるが、operations 層は tool result として読み込まれるため注意重みが低い
- セッションが長くなるほど、作業コンテキスト（デバッグログ、完了済み議論、解決済みエラー等）が蓄積し、ルールの相対的重みが低下する

### 原因構造

問題は Li+ の内容不足ではなく、コンテキストウィンドウの構造的制約に起因する。

1. **注意の薄まり** — 周囲ノイズの増加で tool result 内のルールへの注意重みが下がる
2. **注入ポイントの格差** — system prompt > system-reminder > tool result の優先度階層が存在し、operations 層は最も低い tool result に入る
3. **ゴミの蓄積** — LLM は不要コンテキストを能動的に削除できない（構造的制約）

---

## 解決アプローチ: コンテキスト駆動エージェント分離

動機はコンテキスト最適化であり、手段としてエージェント分離を用い、基盤は GitHub とする。

### メインエージェント（Model Layer + Task Layer）

- Li+core.md + Li+github.md を常駐させる
- 人間との対話による要件蒸留を担当する
- issue 管理は対話から生まれる設計判断を反映する
- Character_Instance による対話品質を維持する
- 独立した修正範囲を特定し sub-issue 化する（要件分割）

### サブエージェント（Operations Layer 実行ワーカー）

- Li+operations.md を毎回新鮮なコンテキストで読む
- 毎回生まれたてのコンテキストであるため注意の薄まりが発生しない
- モデルは同一であり能力差はない
- 成果物は仕様・実装・テストの三つで一つの変更単位とし、分離しない

### 並列実行パターン

GitHub を連携基盤として活用する。

```
メイン: 要件蒸留 → 修正範囲を特定 → sub-issue化
サブA (worktree): sub-issue 1 → branch → push → PR → CI
サブB (worktree): sub-issue 2 → branch → push → PR → CI
メイン: PR結果確認 → マージ判断
```

- サブ同士は連携不要（各自が issue を読んで operations 層に従う）
- GitHub が連携基盤: issue = 共有メモリ、PR = メッセージパッシング、CI = 検証、マージ = コンセンサス
- 確率モデルの揺れは PR 単位で閉じる
- コンフリクトは GitHub が検出し、CI が独立性を検証する

### 段階的検証

1. **パターン1（直列委任）**: メイン → サブ1体に operations 作業を委任 → 結果返却
2. **パターン2（並列委任）**: メイン → 複数サブに worktree 並列で委任 → PR/CI 経由で統合

---

## 制約

- 環境非依存であること（Claude Code / Codex どちらでも使える）
- Li+ の層構造と矛盾しないこと
- 仕様・実装・テストの三つ一組は分離しない（循環依存のため）

---

## 課題

- サブエージェントの往復レイテンシ
- サブの実行結果をメインが受け取る際の受け渡し精度
- サブに Li+ の全体コンテキストがない（prompt で必要分を渡す必要がある）
- 独立した修正範囲の判断精度（メインが間違えるとマージで壊れる。CI が拾う）

---

## 進化

再構築・削除・最適化はすべて許容する。構造の一貫性のみ維持する。
