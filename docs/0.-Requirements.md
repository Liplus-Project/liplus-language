# Li+プログラム 要求仕様書

本文書は、Li+プログラム（Li+core.md, Li+github.md, Li+operations.md, Li+agent.md）が満たすべき要件を定義する。

実装方法はプログラム仕様書（B〜D, H）に記述する。本文書が定めるのは「何を満たすか」であり「どう実装するか」ではない。

---

## 1. Li+言語としての要件

### 1.1 Li+ language と Li+ program を分離して扱える

- Li+ language を高級言語より上位のレイヤーに立つ最高級プログラム言語として扱うこと
- Li+ program を Li+ language を実行する実行系として扱うこと
- Li+ program をAIエージェント本体ではなく、その上で動くオーケストレーション層として扱うこと
- Li+ program は接続済みAIエージェントに対して、レイヤーの接続順・判断基準・再適用条件を与えること
- Li+はプロンプト単体ではなく、継続的な挙動統治の仕組みとして扱うこと

### 1.2 レイヤーで振る舞いを統治する

- 要求レイヤー、コアレイヤー、運用レイヤー、接続レイヤーを役割と見え方の違いとして区別できること
- 優先順は同一レイヤーの内部にだけ定義し、別レイヤー同士を勝敗関係として扱わないこと
- レイヤー間は接続順と依存関係で統合し、別レイヤーの矛盾を「上位レイヤーが勝つ」で解釈しないこと
- タスクの方法は共有された Li+ プログラム全体に保持し、レイヤー分離によって所有関係へ再分配しないこと
- レイヤーごとに責務と再読込・再適用の条件を持てること

### 1.3 要求仕様書がコードである

- 人間の要求から蒸留され、固定された要求仕様書をLi+言語のコードとして扱うこと
- issueの3項目（目的・前提・制約）を収束先の最小構文として扱うこと
- issue本文に専用の完了条件欄を持たず、完了判断は issue 状態と PR/CI/release flow で扱えること
- issueは未完成なメモから開始できること
- 完全版の要求仕様書は `docs/` 配下で管理すること

### 1.4 成果物の三位一体

- 要求仕様書・対象プログラム・CIテストを同じ変更単位でそろえること
- 要求仕様書は「何を正しいとみなすか」を固定する
- 対象プログラムは「要求を現実の動作へ変える」
- CIテストは「変更が要求どおりか」を継続的に観測する

### 1.5 外部記憶としての文書

- issue、docs、commit messageを判断の履歴と根拠の外部記憶として扱うこと
- セッションが変わっても、別のAIが読んでも、同じ要求から近い判断に収束しやすい状態を維持すること

## 2. 対話型コンパイラとしての要件

### 2.1 コンパイルフロー

- 人間がコンパイル開始を承認すること
- Li+AI が要求仕様書を読み、対象プログラムの実装へ落とすこと
- 不足があれば人間に聞き返すこと（コンパイルエラー：仕様の情報不足）
- AIの能力で実装できない場合は人間に返すこと（コンパイルエラー：AIが実装できない仕様）

### 2.2 自己修正

- CIテストを通して失敗を自分で受け取り、修正し、再実行すること
- 自己修正で解消できないときのみ人間へ返すこと
- ループ安全閾値を超えた場合は自動修正を停止し、外部化して人間に委ねること

### 2.3 人間の役割

- 人間は最終判断者であること
- リリースは常に人間の確認が必要であること
- 人間が「待って」「止めて」と言ったら即座に停止すること

---

## 3. 正しさの定義

### 3.1 観測可能な現実

- 正しさは観測可能な現実の挙動によってのみ定義すること
- 説明・意図・内部一貫性は正しさの根拠にしないこと

### 3.2 構造は方法である

- 構造は挙動の安定化機構として使うこと
- 構造のための構造を作らないこと
- 人間とAIの判断をそろえ、現実を安定して観測するために構造を使うこと

---

## 4. 役割分離（ツール非依存）

- 特定のサービスやツールに依存しないこと
- 以下の役割が分離されていれば基盤は何でもよい：
  - Li+プログラム：レイヤー境界・層内順序・行動規則・再適用条件を定義する
  - AIエージェント：要求仕様書・対象プログラム・CIテストを生成し、ツールを実行し、自己修正する
  - バージョン管理：履歴と差分を残す
  - CI/CD：AIが安全に失敗し、観測できる環境
  - 人間：最終判断者

---

## 5. コア挙動の要件

### 5.1 出力の一貫性

- すべての人間向け出力にスピーカー名を付与すること
- 匿名出力を禁止すること
- スピーカーの個性は最初から設定されており、対話はそれを洗練するが置き換えないこと

### 5.2 対話の整合性

- 精度は対話の中で達成すること（対話を上書きする形で達成しないこと）
- キャラクターや前提のドリフトを検知したら再適用してから続けること
- 人間の認知負荷を下げることを最優先とすること

### 5.3 出力密度の制御

- 目標は精度であり、完全性ではないこと
- 過剰説明、網羅的列挙、防御的説明、未来分岐の先読みを避けること
- 概念的な展開は最大3ステップまでとすること（自動化・API操作は例外）

### 5.4 ループ安全機構

- 会話：同じアプローチを2回繰り返したら停止し切り替えること
- タスク/デバッグ：同じアプローチを3回繰り返したら停止し切り替えること
- 切り替え先：視点・表現・媒体・アプローチのいずれかを変えること
- 収束しない場合は強制的な結論を出さず停止すること
- 禁止ループ：説得・感情・過剰最適化・正当化

### 5.5 受容済み判断の扱い

- 人間が受け入れた制約は `accepted constraint` として扱い、ブロッキング集合から外すこと
- 新事実・前提変更・人間の再検討要求がない限り、同じ根拠で再ブロッキングしないこと

### 5.6 レビュー出力の分離

- レビュー・指摘・リスク整理では `now` / `later` / `accepted` に分離すること

### 5.7 ペアレビュー

- 構造的変更を伴うタスクではデュアルレビューループを実行すること
- 収束したらコミットすること

## 6. GitHub運用の要件

### 6.1 作業の起点

- すべての作業はissueから始めること
- issue番号のないcommit / PRを禁止すること
- 関係のないissueを流用しないこと
- issue本文を「現在の要求スナップショット」として扱うこと
- issue作成時に3項目がすべて埋まっていることを要求しないこと
- 話題が永続化すべき作業単位になった時点で、AIが明示指示を待たずにissueを作成できること
- 人間が「issueから始めて」と起動句を言わなくてもissue化できること
- 実装開始前に、実装対象のissue本文が目的・前提・制約へ収束できること
- issueの完了判断は issue 状態と PR/CI/release flow で管理できること
- 収束後も新しい受理情報が入ればissue本文を再構築して更新できること
- 現在地の把握にコメント履歴の読込を必須にしないこと

### 6.2 言語レイヤー分離

- タイトル：ASCII英語のみ（識別レイヤー）
- ボディ：日本語（意味レイヤー）+ issue番号
- 日本語タイトル・英語のみボディを禁止すること

### 6.3 ブランチ管理

- 着手意思は対話の空気から判断すること（NOW / SOON / SOMEDAY）
- trigger modeでもissueの作成・更新とブランチ準備は待機対象にしないこと
- trigger modeでは実装開始とPRレビューを人間トリガーとすること
- `gh issue develop` でブランチとissueをリンクすること
- sub-issueにはブランチを作らないこと
- 同時タスクは親子issue構造を使うこと

### 6.4 CIループ

- PR作成直後にCIループを自動起動すること
- CIループが完了するまでPRタスクは完了しないこと
- CIチェック完了待機の前にマージ可能状態を確認すること
- コンフリクト検知時は自動rebaseを試み、失敗した場合は人間にエスカレーションすること
- CI失敗時はループ安全閾値（3回）内で自動修正を試みること
- 閾値を超えた場合は外部化して人間に委ねること

### 6.5 マージとリリース

- auto-mergeを使用すること
- リリース作成前にCD完了を確認すること
- AIが作成するリリースは必ずプレリリースとすること
- バージョン種別の判断は人間が行うこと

### 6.6 マイルストーンとラベル

- すべてのissueに作成時点でマイルストーンとタイプラベルと成熟度ラベルを付与すること
- マイルストーンはリリース単位とすること
- ライフサイクルラベルは状態変化時に適用すること
- 成熟度ラベルはissue本文の収束度に応じて更新すること
- 詳細な運用ルールはLi+operations.mdに定義すること

### 6.7 ドキュメント同期

- `docs/` を正本とすること
- 実装変更を含むPRは対応する `docs/` の更新を同一PR内に含めること
- 実装前に要求仕様書を先に作成または更新すること
- Li+ の挙動・運用判断は standalone ADR だけに閉じず、番号付き要求仕様と対応する運用文書に固定すること
- 要求仕様書は必要になるまで `0.-Requirements.md` に集約してよく、分割が可読性を上げる時だけ `1` 以降へ展開してよい

## 7. セッション初期化の要件

### 7.1 ブートストラップ

- セッション開始時にLi+config.mdを読み込み、実行すること
- ホスト環境の指示ファイル（CLAUDE.md / AGENTS.md）にCharacter Instanceを含めること
- 配布先workspaceでは、会話の基本言語と成果物のプロジェクト言語を分離できること
- これらの既定値は Li+config.md に保持し、未設定ならセッション開始時にAIが対話で確定して書き戻せること
- 人間の明示指示で、現在の返答または成果物の言語を上書きできること
- このworkspace言語契約は liplus-language リポジトリ内部の運用言語と分離されること

### 7.2 セッション継続性

- コンテキスト圧縮・再開・セッション継続時にLi+core.mdとLi+github.mdを再読込すること
- トリガーベースの再読込：issue、branch、commit、PR、merge、release の各操作時に、Li+github.md または Li+operations.md の該当セクションを再読込すること
  - runtime=claude: ランタイムhook（PostToolUse / UserPromptSubmit）により強制発火すること
  - hook定義はLi+claude.mdに保持し、Li+config.md Step 6で初回生成すること
- Li+operations.mdはオンデマンドで読み込むこと（毎セッション必須ではない）

### 7.3 前景Webhook通知の取り込み

- ホストが各ターン先頭でローカル確認を実行できる場合、前景スレッドが軽量なGitHub webhook通知を確認できること
- 前景Webhook確認は内部 housekeeping として無言で実行し、確認中であることや empty/no-op 結果を人間向け出力へ混ぜないこと
- runtime=claude: UserPromptSubmit hookによりユーザーメッセージ毎にローカルwebhookストアの pending_count を自動チェックすること
- 通知源の優先順位は `mcp__github-webhook-mcp` → ローカル webhook ストア → 利用不可 の順であること
- ローカル fallback は shared instruction に機械固有の絶対パスを埋め込まず、設定値またはワークスペース相対の候補から解決できること
- 通知源が見つからない場合は黙ってスキップできること
- 前景へ通知として出したローカル pending イベントは、その場で consume し、関連生成物も削除できること
- この取り込みフローでは別AIプロセスを起動せず、前景スレッドが主導して扱うこと

## 8. 進化の要件

- Li+プログラムの再構築・削除・最適化はすべて許容すること
- 構造の一貫性のみ維持すること
